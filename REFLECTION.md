# REFLECTION — Бие даалт 14

## 1. Аль assertion хамгийн их үнэ цэнэтэй санагдсан вэ? Яагаад?

Хамгийн их үнэ цэнэтэй гэж схемийн (JSON schema / property) баталгаажуулалтыг тооцлоо. Status code 200 буцаах нь API ажиллаж байгааг харуулдаг ч response-ийн агуулга зөв эсэхийг баталгаажуулдаггүй. Жишээ нь `pm.expect(d).to.have.property("id")` болон `pm.expect(d.email).to.match(/@/)` зэрэг тестүүд нь API-ийн contract-ыг шалгадаг. Хэрэв API өөрчлөгдөж `email` талбарын нэр өөр болбол энэ тест шууд fail болж, frontend эсвэл бусад service-д нөлөөлөхөөс өмнө асуудлыг илрүүлнэ. Энэ нь "API ажиллаж байна" биш "API зөв өгөгдөл буцааж байна" гэдгийг баталгаажуулдаг учраас хамгийн чухал гэж үзлээ.

## 2. Negative test жишээгээ дэлгэрэнгүй тайлбарла

`GET /users/999999` request нь байхгүй хэрэглэгч хайх negative test юм. Энэ тест 404 status code буцаахыг шалгадаг. Зорилго нь API зөвхөн амжилттай замаар биш алдааны замаар ч зөв ажиллаж байгааг баталгаажуулах явдал. Жишээ нь хэрэв API 404-ийн оронд 500 буцаавал серверт дотоод алдаа байна гэсэн үг бөгөөд энэ тест тэрийг илрүүлнэ. Мөн response time 1 секундээс доош байгааг ч шалгасан — алдааны үед ч API удаашрахгүй байх ёстой. Negative test нь "хэвийн ажиллах"-ыг шалгахаас илүү чухал байдаг, учир нь алдааны зам бол мөн API-ийн contract-ийн хэсэг.

## 3. Postman дотор амжилттай ажиллаж байсан тест Newman-д fail болсон уу?

Postman дотор бүх тест амжилттай ажиллаж байсан. Гэхдээ Newman-д анх ажиллуулахад `env.ci.json` файлд шаардлагатай хувьсагчид (`createdPostId`, `chainedPostId`) байхгүй байсан учраас chain хийсэн request-үүд fail болох аюул байсан. `env.dev.json` болон `env.ci.json`-ийн хувьсагчдын бүтэц ялгаатай байсан нь асуудал үүсгэж болох байсан. Үүнийг `env.ci.json`-д шаардлагатай бүх хувьсагчдыг нэмж шийдсэн. Энэ нь орчны тохиргоо (environment configuration) хэр чухал болохыг харуулсан сайн жишээ болсон.

## 4. Token эсвэл secret-ыг хэрхэн зохицуулсан вэ?

JSONPlaceholder нь auth шаардахгүй public API учраас token хэрэглээгүй. Гэхдээ env файлуудад `baseUrl`-ыг environment variable болгон хадгалсан нь зөв практик. Хэрэв token шаардагдах байсан бол `env.dev.json`-д `"value": "REPLACE_THIS"` гэсэн placeholder ашиглаж, бодит утгыг GitHub Secrets-т хадгалах байсан. `env.ci.json`-д placeholder ашигласнаар token GitHub-т шууд commit хийгдэхгүй байх аюулгүй байдлыг хангасан. Secret-ыг хэзээ ч `.json` файлд шууд бичиж commit хийх ёсгүй.

## 5. API өөрчлөгдвөл collection-ийн аль хэсэг хамгийн их эвдрэх вэ?

Schema болон property шалгадаг тестүүд хамгийн эмзэг. Жишээ нь `pm.expect(d).to.have.property("email")` тест нь API-ийн response-ийн талбарын нэр өөрчлөгдвөл шууд fail болно. Мөн chain хийсэн request-үүд эмзэг — `createdPostId` хувьсагч зөв set хийгдэхгүй бол дараагийн бүх request fail болно. Үүнийг бууруулахын тулд: нэгдүгээрт, schema validation-ыг тусдаа файлд гаргаж, API өөрчлөгдөхөд зөвхөн тэр файлыг засах; хоёрдугаарт, chain хийсэн хувьсагчдад default утга өгөх; гуравдугаарт, collection-ыг тогтмол Newman-аар ажиллуулж regression илрүүлэх нь зүйтэй.
